### Analytical focus (general)

When reading papers, always look at:

- System model assumptions and simulation parameters
- Baseline selection (are the right baselines compared?)
- Whether the stated motivation aligns with the chosen methodology
- Whether experimental scope validates the paper's claims (e.g. claims scalability but only tests one node)
- Data-credibility red flags (see the **Review mechanism** section below)

### Response style

- Reply in the preferred output language specified above.
- Keep established technical terms in their original form (do not translate loanwords like "handover", "offloading", "reward function").
- Analysis must be concrete: when naming a weakness, cite the section/table/figure and state exactly why it is one.

# LLM Wiki — shared schema

Below is the universal workflow spec, consistent with all ten slash commands under `.claude/commands/`.

## Overview

This vault is an LLM-maintained research wiki. The user reads, curates, and asks questions; the LLM summarizes, cross-references, and maintains page linkage. The pattern is adapted from Andrej Karpathy's [LLM wiki gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f).

## Directory structure

```
vault/
├── raw/                    # Immutable source material (read-only for the LLM)
│   ├── papers/             # Paper PDFs or extracted text
│   └── web/                # Web clippings
├── wiki/                   # LLM-compiled knowledge pages
│   ├── index.md            # Master index (Dataview-generated)
│   ├── log.md              # Operation log (reverse chronological)
│   ├── dashboard.md        # Health-check queries
│   ├── concepts/           # One page per technical concept
│   ├── entities/           # Research teams, standards, tools, algorithms
│   ├── sources/            # One page per paper/article
│   └── synthesis/          # Cross-source comparative analyses
```

## Page formats

`wiki/index.md` is generated by the **Dataview** plugin. Every wiki page must have a `summary:` frontmatter field (source pages also need `venue_short:`) so that Dataview renders the index correctly.

### Source page (`wiki/sources/`)

```markdown
---
title: "Full paper/article title"
authors: [author list]
venue: "Full venue name (journal/conference/site)"
venue_short: "Short venue tag (e.g. IEEE IoT-J, arXiv, JNCA)"
summary: "One-line headline for the index"
year: 2025
tags: [domain-tags]
raw: "[[raw/papers/filename.pdf]]"
ingested: 2026-04-10
---

# Title

## Core contribution
(2-3 sentences: what it does and what problem it solves)

## Method
(Key technical details: algorithm, model, assumptions)

## Key findings
(Experimental results, numbers, conclusions)

## Relation to the knowledge base
- Supports the position in [[concepts/some-concept]] by ...
- Differs from [[sources/another-paper]]'s method in ...
- Fills the gap mentioned in [[concepts/some-concept]]

## Personal notes
(Ideas, questions, and reactions raised during discussion; optional)
```

### Concept page (`wiki/concepts/`)

```markdown
---
title: "Concept name"
aliases: [alt-name-1, alt-name-2]
tags: [category-tags]
source_count: 3
summary: "One-line summary for the index"
last_updated: 2026-04-10
---

# Concept name

## Definition
(Clear technical definition, integrating multiple sources in your own words)

## Current state
(Prevailing consensus and mainstream approaches)

## Key methods and comparison
(Different approaches proposed by different papers, with trade-offs)

## Open problems
(Unsolved challenges worth further exploration)

## Related concepts
- [[concepts/related-A]]
- [[concepts/related-B]]

## Sources
- [[sources/paper-1]] — proposed X
- [[sources/paper-2]] — refuted Y, proposed Z
```

### Entity page (`wiki/entities/`)

For research teams, standards bodies, tools, and notable algorithms relevant to your domain. Compact format: definition, relation to research, related sources. Frontmatter: `title`, `type`, `tags`, `summary`.

```markdown
---
title: "Entity name"
type: "tool | standard | team | algorithm | ..."
tags: [category-tags]
summary: "One-line summary"
---
```

### Synthesis page (`wiki/synthesis/`)

For cross-source comparisons, literature reviews, and trend analyses. Usually promoted from a query answer worth preserving. Frontmatter: `title`, `aliases`, `tags`, `source_count`, `created`, `summary`.

```markdown
---
title: "Synthesis title"
aliases: [alt-names]
tags: [synthesis, topic]
source_count: 8
created: 2026-04-18
summary: "One-line summary"
---
```

## Review mechanism (review gate)

The review gate prevents auto-generated paper notes — which may contain summarization errors, wrong numbers, or misunderstood concepts — from propagating into the wiki's concept, source, and synthesis pages and contaminating the knowledge base.

### Core design principles

1. Every paper note must pass human review before it can be cited by wiki pages.
2. Notes produced by `/read-paper` default to `unreviewed` (untrusted).
3. Downstream workflows must check `review_status` before using a note.
4. Review findings must leave an audit trail (for traceability and pattern-learning).

### `review_status` frontmatter

Applies to all `.md` paper notes at the vault root (those tagged `paper`).

```yaml
---
title: "..."
# ... other fields ...
review_status: unreviewed    # required: unreviewed / reviewed / reviewed_with_notes / disputed
reviewed_date: null          # date the review was completed; null while unreviewed
reviewed_by: null            # the reviewer; null while unreviewed
review_notes: null           # one-line summary of findings; may be null for clean reviews
---
```

### State definitions

| State | Meaning | Can downstream cite? | Trigger |
|---|---|---|---|
| `unreviewed` | Just produced by `/read-paper`, not yet audited | No — `/ingest-to-wiki` blocks; other skills soft-warn | Default |
| `reviewed` | User has verified every claim against the PDF | Yes, freely | After `/review-note` completes cleanly |
| `reviewed_with_notes` | Reviewed, but minor caveats recorded | Yes, but the citation must carry the caveat | After `/review-note` finds minor issues |
| `disputed` | Major problems found (data-credibility issues, method/experiment mismatch, etc.) | Only with explicit dispute marking on the wiki source page | After `/review-note` finds major issues |

### Downstream review-gate checks

All skills that read a root-level paper note and propagate its content into the wiki must check `review_status` before running:

| Skill | When it checks | How it handles `unreviewed` |
|---|---|---|
| `/ingest-to-wiki` | When reading a paper note | `[HARD GATE]` — stop immediately, require `/review-note` first |
| `/paper-summary` | Before editing the note | `[SOFT WARN]` — warn that output may be wrong; user may continue |
| `/paper-report` | Before editing the note | `[SOFT WARN]` — warn and prepend a note that the report is based on an unreviewed paper |
| `/concept-extract` | Before extracting concepts | `[SOFT WARN]` — warn; produced concept pages get `status: draft` |
| `/organize-vault` | Before building links | `[SOFT WARN]` — links to unreviewed notes carry an `[UNREVIEWED]` marker |
| `/read-paper` | N/A (producer) | Must set `review_status: unreviewed` on the produced note |
| `/review-note` | N/A (this is the review flow) | — |

### Review flow (`/review-note`)

Triggered by `/review-note <filename>` or by saying "review this paper". The review is a full-depth audit, not a skim:

1. Read the PDF + the current note
2. Decompose the note into all claims (motivation, model assumptions, method steps, formulas, experimental setup, every quantitative data point, conclusions)
3. Check each claim against the PDF:
   - `[OK]` consistent — mark it and move on
   - `[DIFF]` inconsistency — list the difference, let the user decide
   - `[FAIL]` error — note the correct value
   - `[?]` not verifiable from the PDF — ask about the source
4. Check logical consistency across six axes:
   - Motivation → problem definition
   - Problem definition → methodology
   - Methodology → experiment
   - Experiment → results
   - Results → conclusions
   - Related work → baselines used
5. Check data-credibility red flags:
   - Perfect arithmetic progressions in result tables (real simulations almost never produce this)
   - Fixed offsets across models (e.g. Accuracy − Precision always exactly 0.03)
   - Proposed method winning on every single metric (real methods have trade-offs)
   - In-text narrative contradicting the table numbers
   - Experimental scope vs. motivation mismatch (claims scalability but tests only a small setup)
6. Produce a Review Checklist and ask the user to confirm each item
7. Update the note:
   - Edit paragraphs the user confirmed should change
   - Append a `## Review log` section with date, reviewer, findings, and final state
8. Update frontmatter: `review_status`, `reviewed_date`, `reviewed_by`, `review_notes`
9. Append an entry to `wiki/review-history.md`

### Cascade check after a review

If a note turns out to have errors and is already cited by wiki pages, re-check those pages after the review:

1. Fix the root-level note first.
2. Grep `wiki/` for `[[note-name]]` to find all citations.
3. For each citing page, decide whether it needs a fix:
   - Wiki source page: usually must be fixed (sync the core-contribution and key-findings sections)
   - Wiki concept page: if the cited number was wrong, either fix it or add a warning marker
   - Wiki synthesis page: check whether the error affects the conclusion
4. Log each cascade edit to `wiki/log.md` with the tag `review-cascade`.

### Cascade caveat rules (on citations)

When a wiki page cites a source whose `review_status` is not `reviewed`, the citation must carry a matching caveat:

| Cited source status | Required marker on the citation |
|---|---|
| `reviewed` | None — plain citation |
| `reviewed_with_notes` | `[[sources/x]] — proposed Y (note: see the Z caveat in review_notes)` |
| `disputed` | `[[sources/x]] — proposed Y ([DISPUTED]: reason; borrow the idea, not the numbers)` |
| `unreviewed` | `[[sources/x]] — proposed Y [UNREVIEWED: produced by /read-paper, PDF not yet audited]` |

This is what `/lint-wiki` audits in its Phase 2 pass. The review gate only has value if its conclusions propagate to every citation site; otherwise a disputed paper ends up cited three pages deep as if it were fact, and the wiki becomes a laundromat for bad claims.

### Exception: classical papers or scanned PDFs

The "compare each claim against the PDF" rule can be relaxed, with explicit user authorization, when **all** of the following hold:

1. The content is a well-established classical definition, theorem, or formula (e.g. Shannon entropy, Diffie-Hellman, Big-O notation)
2. The PDF cannot be auto-extracted (scanned, complex layout, or missing)
3. The note only restates the classical definition, without author-original claims
4. The user explicitly authorizes the simplified review

In this case, verify against the LLM's internal knowledge of the classical definition instead of the PDF, and record "verified against internal knowledge, not the PDF" in `review_notes` and `wiki/review-history.md`. This exception does not apply to papers with original experimental data or methods.

### Review meta pages

Two live pages in `wiki/` surface the review state. Both are personal (gitignored in this repo) but the convention is part of the workflow:

- `wiki/pending-review.md` — a Dataview page listing every root-level note with `review_status: unreviewed`. Acts as a queue of papers awaiting `/review-note`. Open it to pick the next paper to audit.
- `wiki/review-history.md` — append-only audit log. Every `/review-note` execution appends a reverse-chronological entry with findings and the final status.

Neither file needs manual maintenance once Dataview is installed and the review flow is in use.

## `index.md` format

`index.md` is auto-generated by Dataview. Users no longer edit index tables by hand — as long as every page's frontmatter is correct, the queries below include it automatically.

```markdown
# Knowledge base index

\`\`\`dataview
TABLE WITHOUT ID file.folder AS "Section", length(rows) AS "Pages"
FROM "wiki/concepts" OR "wiki/entities" OR "wiki/sources" OR "wiki/synthesis"
GROUP BY file.folder
SORT file.folder ASC
\`\`\`

## Concepts

\`\`\`dataview
TABLE WITHOUT ID file.link AS "Page", summary AS "Summary", source_count AS "Sources"
FROM "wiki/concepts"
SORT source_count DESC, file.name ASC
\`\`\`

## Sources (newest first)

\`\`\`dataview
TABLE WITHOUT ID file.link AS "Page", summary AS "Title", venue_short AS "Venue", year AS "Year"
FROM "wiki/sources"
SORT year DESC, file.name ASC
\`\`\`

## Entities

\`\`\`dataview
TABLE WITHOUT ID file.link AS "Page", type AS "Type", summary AS "Summary"
FROM "wiki/entities"
SORT type ASC, file.name ASC
\`\`\`

## Synthesis

\`\`\`dataview
TABLE WITHOUT ID file.link AS "Page", summary AS "Topic", source_count AS "Cited sources"
FROM "wiki/synthesis"
SORT created DESC
\`\`\`
```

Keep a `<details>`-wrapped static fallback at the end of `index.md` for plain-text viewing when Dataview is not loaded. Mark the fallback as "non-authoritative".

## Operations

### 1. Ingest

Triggered by `/ingest-to-wiki` or a natural phrase like "integrate this into the wiki".

Steps:

0. `[HARD GATE]` Review check: if the source is a root-level paper note, read its `review_status`. If `unreviewed`, stop and require `/review-note` first. If `disputed`, require explicit user confirmation to continue. If `reviewed_with_notes`, continue but record the caveat on the wiki source page.
1. Read the source (from `raw/` or from the conversation)
2. Create a source page under `wiki/sources/`. Frontmatter must include `summary:` and `venue_short:`.
3. Scan `wiki/index.md` (the Dataview-rendered output) or grep for related existing pages.
4. Update or create related concept pages (`wiki/concepts/`). New concept pages must have `summary:` and `source_count:`.
5. Create or update entity pages as needed. New entity pages must have `summary:` and `type:`.
6. `index.md` updates itself via Dataview; just make sure the new page's frontmatter is complete.
7. Prepend a new entry to `wiki/log.md` (reverse-chronological): the new entry goes at the top of the log, right after the convention header block and before the most recent existing `## [...]` entry.
8. Report what was created/updated and any new associations or contradictions.

Notes:
- Process one source at a time (batching degrades review quality).
- When updating a concept page, **merge** rather than overwrite — preserve prior sources' positions and mark disagreements.
- If a new source contradicts existing wiki claims, mark the disagreement on the concept page.
- Missing frontmatter fields cause Dataview to render blank cells; always fill them.

### 2. Query

Triggered by the user asking a research question.

1. Read `wiki/index.md` (Dataview tables) to locate relevant pages, or grep for a specific tag/keyword.
2. Read the relevant concept and source pages.
3. Answer synthesizing across pages, citing with `[[wiki-links]]`.
4. If the answer produces a valuable new analysis (comparison table, trend summary, novel insight), ask the user whether to archive it as a `wiki/synthesis/` page.

Query tool hierarchy:

- Plain text or regex matches: use the Grep tool directly.
- Structured queries (frontmatter filters, backlinks, orphan detection, vault stats): use `/search-vault`. Six modes — `structured`, `content`, `orphan`, `inbound`, `stats`, `semantic` — all read-only. See `.claude/commands/search-vault.md` for the full spec.
- Meaning-based lookup: `/search-vault semantic` wraps the optional qmd backend (BM25 + vector + LLM rerank). qmd must be installed separately.

### 3. Lint

Two lint commands cover different concerns. Both are read-only audits.

#### 3a. `/lint-wiki` — wiki structure and cascade audit

Triggered by `/lint-wiki` or "check the wiki". Recommended every 5–10 new sources.

Checks:
- Orphan pages (no inbound `[[link]]`)
- Concepts mentioned in prose but not yet promoted to their own page
- Source pages missing the "Relation to the knowledge base" section
- Concept pages whose "Current state" or "Key methods" sections need updating after a new source joined
- Frontmatter field completeness (Dataview depends on this)
- `updated:` vs `last_updated:` naming inconsistency
- Cascade caveat consistency — every citation of a `disputed`, `reviewed_with_notes`, or `unreviewed` source must carry the correct marker
- Contradictions or outdated claims
- Suggested next reading directions

Output: append a lint entry to `wiki/log.md`.

#### 3b. `/lint-related-work` — Related Work framing fairness

Triggered by `/lint-related-work <path-to-note-or-pdf>`. Recommended right after `/read-paper`, and again on the primary baselines before preparing a seminar talk.

Phase 1 (mechanical): grep for absolutist wording (`completely overlooked`, `entirely fails`, …), list every baseline (`Author et al. [N]`), and flag scope mismatches (e.g. an authentication paper citing a task-scheduling paper as a baseline).

Phase 2 (optional, needs the baseline PDF in `raw/papers/`): for each baseline criticised with absolutist wording, compare the paper's criticism against the baseline's own Abstract and Contributions. Surfaces self-contradictory criticisms (paper calls baseline "centralized" while the baseline's Abstract says "distributed").

Output: a structured markdown report. When issues are found, append one entry to `wiki/log.md` (`lint-related-work | <paper> -- N red flags`) and suggest — never auto-apply — a `review_status` downgrade. See `.claude/commands/lint-related-work.md` for the full spec and the starter dictionary of absolutist wording.

### 4. Review

Triggered by `/review-note <filename>` or "review this paper". New notes from `/read-paper` default to `unreviewed` and must pass review before `/ingest-to-wiki` will touch them. See the full flow in **Review mechanism** above.

### 5. Slide draft (seminar material)

Triggered by `/paper-ppt` or "build the slides for X".

1. Review-gate check (`disputed` mandates a warning slide; `unreviewed` warns and prepends a note).
2. Read the paper note + PDF.
3. Lay out 14–18 slides: Title, Outline, Background, Gap, Related Work, System Model, Method, Experiment, Results, Pros & Cons, Relation to My Research, Conclusion, Q&A.
4. Language rule: **slide bullets** in English (≤15 words each), **speaker notes** in the user's preferred output language.
5. Exception slides (no bullets): title, architecture (ASCII/figure), data table, math formula.
6. Output: `slide-draft-{paper-slug}-{YYYY-MM-DD}.md` at vault root.
7. Append a "Related links" section pointing to the paper note, wiki source page, related concepts/synthesis, and review history.

Full spec in `.claude/commands/paper-ppt.md`.

### 6. Synthesis

Triggered when the user asks for cross-source comparison, or when a query answer is worth promoting.

Goal: produce a page under `wiki/synthesis/` rather than another per-source summary.

Rules:
- Frontmatter must include `source_count`, `created`, `summary`.
- All cited sources must be `reviewed` or `reviewed_with_notes` — synthesis cannot build on `unreviewed` or `disputed` data.
- Structure is flexible but should cover: scope, core question, comparative analysis, open problems, related pages, cited sources.
- List all cited sources at the end for traceability.

## Writing principles

- Use the output language specified in the Researcher profile above.
- Keep technical terms in their original form.
- Be precise and grounded — avoid generic AI-tone filler.
- Use `[[wiki-links]]` liberally — concepts should be interlinked, and source-to-concept linking is mandatory.
- Every page needs YAML frontmatter for Dataview queries to work.
- If a source contains an important figure, remind the user to save it locally via Obsidian's attachment-download feature.
